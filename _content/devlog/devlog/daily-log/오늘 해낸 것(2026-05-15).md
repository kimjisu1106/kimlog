## 오늘 한 일

- 회사 툴
	- 시설물별 사양(Spec 항목)을 인라인 테이블로 관리.
		- 이유: 별도의 field로 따로 관리하기에는 시설물별 spec이 너무 다양했다.
		- 해결: 항목은 사용자가 추가/삭제 가능하도록 하되 기존에 입력했던 것을 목록으로 보여줘 고를 수도 있게 했다. 문제점은 사용자가 항목을 오타낼 수 있다는 점인데 일단 사용자가 적은 상태이므로 패스
	- 시설물의 관할(purpose) field 생성
		- 이유: 시설물 마다 어디를 관할하는 것인지 표기가 필요했음
		- 해결: text field 생성
	- work에서 연관 시설물 선택 시 location 자동 입력. 
		- 이유: 시설물 model에 위치가 있는데 시설물을 수리하는 work의 특이사항이 없다면 시설물이 있는 위치에서 수리할 것이므로 재차 입력하기 번거로웠다
		- 해결: 혹시 수동 입력 필요할 수 있으니 boolean 처리
	- Payment와 Work 관계를 CASCADE로 변경.
		- 이유: 작업을 했으면 돈이 나갈텐데 돈이 안나간다면 작업을 하기로 했다가 안한 것이고 그런 경우 해당 work를 fk를 사용하는 곳들에 데이터가 남게된다.
		- 해결: 혹시 무상 수리되는 건이 있을 수 있으니 Payment 삭제 시 연결된 work 목록을 이용한 경고 표시하여 분리 해야함을 사용자로 인지하도록 함.
	- ErrorLog 중복 방지
		- 이유: 보통 오류가 발생하면 동일한 동작을 여러번 누르게되어 같은 오류가 여러번 발생함
		- 해결: 동일 (exception_type, exception_message, path) 조합은 get_or_create + F("count") + 1 + last_seen 갱신.
		- 결과: 발생 횟수, 최초/최근 발생 시각 컬럼 생성 및 최근 발생 역순 정렬해서 몇 번 발생되었고 가장 마지막에 언제였는지 확인하도록 구현

## 배운 점

- @login_required는 바로 아래 def에 붙는다.
	- 데코레이터와 함수 사이에 다른 def를 끼워 넣으면 그 함수에 잘못 적용된다. 헬퍼 함수를 @login_required 위에 두거나, 데코레이터 직후에 바로 대상 함수가 오도록 순서를 지켜야 한다.
- Django FK+M2M 혼합 prefetch 주의
	- 문제점: M2M 거쳐 FK, 또는 FK 거쳐 M2M을 prefetch_related로 체이닝하면 불안정하다.
		- Work.facility→ M2M (여러 facility)
		- facility.location → FK (하나의 location)
		- Work.location      ← 여기에 자동으로 채우려는 상황
	- 방안: FK는 select_related, M2M은 prefetch_related → 각각 따로 접근할 것
- ErrorLog 중복 방지: get_or_create + F() 패턴
	- 문제점: 동일 에러가 반복 발생할 때 매번 row를 INSERT하면 로그가 폭증한다. 
	- 방안: get_or_create로 중복 판별 후, 기존 레코드는 filter(pk=).update(count=F("count")+1, last_seen=now())로 처리한다. 
	- get_or_create: Django ORM 기본 제공 함수. 있으면 get, 없으면 create
	- F(): Python에서 계산할 경우 동시에 동일한 에러 발생 시 동시에 count=1를 읽고 count=2로 저장하는 오류를 방지하기 위해 DB 레벨에서 계산한다(log.save()는 race condition에 취약하므로 Python이 계산하고 DB에 저장하는게 아니고 DB에 +1하라는 명령만 보내는 방식). 
- auto_now_add=True는 기존 모델에 그냥 추가할 수 없다.
	- 이유: ErrorLog 최근 발생 순으로 정렬하기 위해 field 추가함.
	- 문제점: 기존 row에 값이 없으므로 Django가 default를 요구한다. 
	- 방안: default=timezone.now로 선언하면 마이그레이션 시 기존 row에 현재 시각이 채워지면서 통과된다.
- `_pendingDraftItem` 패턴 — `setTag()` 함수에 추가 데이터 전달하는 방법
	- 이유: Draft를 FK로 참조하는 Payment 생성 시 Draft에서 title, date등을 가져오기 위함.
	- 문제점: setTag()는 id/name만 받는데 Draft 객체 전체가 필요함. 서버에서 처리하는 경우 Payment 생성 후 저장해야만 확인이 가능해서 저장 → 확인 → 수정으로 단계가 길어짐. 
	- 방안: `_pendingDraftItem`에 임시 저장 후 콜백에서 소비하면 생성과 수정이 한 번에 가능. 단, 초기화(edit 폼 로드 시 자동실행 방지)를 위해 null 리셋 타이밍을 정확히 잡아야 한다.
