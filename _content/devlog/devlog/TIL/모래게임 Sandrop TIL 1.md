## 규칙 엔진 (Dart · Python)

### 같은 규칙을 두 언어로 짤 때는 fixtures가 스펙이다

이 게임의 규칙 엔진(모래가 어떻게 쌓이나, 어떤 바스켓을 꺼낼 수 있나)은 두 곳에 있다. 하나는 게임 본체인 Dart, 하나는 레벨을 자동 생성하는 파이썬 도구. 같은 규칙을 두 번 구현한 셈이라, 한쪽만 고치면 둘이 서로 다르게 동작하는 사고가 나기 쉽다.

이걸 막는 게 fixtures다. fixture는 "입력과 기대 출력을 미리 적어둔 테스트 케이스"를 말한다. 규칙을 말로 적은 문서가 아니라, 기계가 그대로 재생해볼 수 있는 데이터. 벽(통과 못 하는 칸) 규칙을 새로 넣을 때 순서는 이렇게 했다.

1. 문서 스펙을 먼저 고친다
2. Dart와 파이썬 양쪽 코드를 고친다
3. 공통 fixture(JSON) 케이스를 하나 추가한다

그러면 Dart 테스트와 파이썬 테스트가 같은 fixture를 각자 돌려 결과가 기대값과 맞는지 대조한다. 둘 중 하나라도 규칙이 어긋나면 그쪽 테스트가 깨지므로, 두 구현이 갈라지는 걸 바로 잡는다.

```json
{
  "name": "case5_walls",
  "board": { "size": [3, 3], "walls": [[1, 0], [1, 1]] },
  "baskets": [[2, 0], [0, 1], [0, 2], [2, 2]],
  "expectSelectable": [[2, 0], [0, 1]]
}
```

한 가지 함정. fixture는 틀렸을 때 틀렸다고 티가 나야 쓸모가 있다. 벽을 무시하고 짜도 우연히 같은 결과가 나오는 배치라면, 벽 규칙이 망가져도 테스트가 통과해버려 회귀(regression, 예전에 고친 버그가 되살아나는 것)를 못 잡는다. 그래서 "벽이 있으면 선택 불가, 없으면 가능"이 확실히 갈리는 배치로 케이스를 짜야 한다.

### 스냅샷 undo는 실시간 시뮬레이션과 그냥은 양립하지 않는다

먼저 용어부터. 스냅샷 undo란, 플레이어가 한 수 둘 때마다 그 직전의 게임 상태 전체(보드·벨트·캔버스)를 통째로 저장(스냅샷)해 뒀다가, 되돌리기를 누르면 그 저장본으로 되돌리는 방식이다. 턴제 게임(한 수 두고 → 결과 확정 → 다음 수)에서는 잘 맞는다. 수와 수 사이에 시간이 안 흐르니까.

그런데 이 게임은 실시간이다. 바스켓을 탭하면 → 엘리베이터를 타고 올라가 → 벨트에 실려 → 도는 동안 조금씩 모래를 붓는다. 이 과정이 시간에 걸쳐 계속 진행된다. 여기에 턴제식 스냅샷 undo를 그대로 옮기면 두 군데서 깨진다.

첫째, "되돌릴 바스켓이 아직 살아 있는가"를 판정할 때. 되돌리기는 "마지막에 올린 그 바스켓"을 보드로 되돌리는 것이라, 그 바스켓이 아직 존재하는지 확인해야 한다. 그런데 코드는 바스켓을 레일(벨트)과 버퍼에서만 찾았다. 문제는 엘리베이터로 올라가는 중인 바스켓 — 이미 보드에서 빠졌지만 아직 벨트엔 안 실린 상태라, 레일에도 버퍼에도 없다. 분명 화면에 살아 있는데 "없음"으로 판정돼 되돌리기가 안 됐다.

둘째, 스냅샷을 찍은 뒤 세상이 계속 움직인다. 스냅샷을 찍고 나서 다른 바스켓이 벨트에 실려 모래를 부었다면, 그 스냅샷은 그 붓기를 모른다(찍힌 시점엔 없던 일이니까). 이 상태에서 되돌리면 그 사이 부어진 모래가 통째로 사라진다. 색별 총 모래량이 캔버스 픽셀 수와 딱 맞아야 클리어되는 게임이라(§3.3 설계 불변식), 모래가 증발하면 그 레벨은 영영 클리어 불가가 된다.

해결의 핵심은 두 가지. (1) 엘리베이터로 이동 중인 바스켓들의 집합(inTransit)을 엔진이 따로 들고 있게 해서 "살아 있는가" 판정에 포함시킨다. (2) 스냅샷을 찍을 때 그 순간의 inTransit 집합도 같이 기록해두고, 되돌릴 때 "지금의 이동 중 집합이 스냅샷 찍을 때와 똑같은가"를 확인한다. 같다는 건 "그 사이 새로 실리거나 부어진 게 없다 = 세상이 안 움직였다"는 뜻이고, 그럴 때만 안전하게 복원한다.

아래 코드는 되돌리려는 통 말고 다른 통이 그새 안 움직였는지 확인한다.

```dart
// ❌ 레일·버퍼만 확인 — 엘리베이터 이동 중이면 살아 있어도 "없음"으로 오판
final alive = rail.any((b) => b?.id == id) || buffer.any((b) => b.id == id);

// ✅ 이동 중(inTransit)도 포함해서 "살아 있는가"를 본 뒤,
//    되돌릴 대상(id)을 뺀 나머지 이동 중 집합이 스냅샷 시점과 똑같을 때만 복원 허용
final alive = onRail || inBuffer || inTransit.contains(id);
final nowTransit = Set<int>.of(inTransit)..remove(id); // 되돌릴 대상은 빼고 비교
final consistent = nowTransit.length == snap.transitIds.length &&
    nowTransit.containsAll(snap.transitIds); // 그 사이 다른 변화가 없었나
```

교훈은, 복원 가능한 조건을 흐릿하게 두지 말고 명시적으로 판정하고, 그 조건을 못 채우면 차라리 기능을 막는 게(버튼 비활성) 총량이 깨진 채로 복원해 레벨을 망가뜨리는 것보다 낫다는 것.

### 서로 다른 두 곳의 조건이 동시에 참일 수 없으면, 기능은 조용히 죽는다

가드(guard)란 "이 조건이 맞을 때만 실행하라"고 앞을 막아서는 검사다. 되돌리기 기능엔 가드가 두 군데 있었다. UI 쪽 가드는 "벨트가 완전히 비어 있어야 버튼 활성", 엔진 쪽 판정은 "되돌릴 바스켓이 벨트에 살아 있어야 실행". 각각 따로 보면 둘 다 말이 된다.

문제는 이 둘이 실시간 벨트에서는 절대 동시에 참일 수 없다는 것. 바스켓이 벨트에 있으면 → 벨트가 안 비었으니 UI 가드가 거짓(버튼 비활성). 벨트가 비면 → 되돌릴 바스켓도 이미 사라졌으니 엔진 가드가 거짓. 두 조건의 교집합이 공집합이라, 버튼은 예외도 로그도 없이 그냥 영원히 눌리지 않았다. 에러가 안 나니 아무도 눈치 못 챘다.

교훈. 가드가 UI·엔진처럼 여러 층에 흩어져 있으면, "이 기능이 실제로 발동될 수 있는 상태가 존재하긴 하나?"를 한 번은 손으로 두 조건을 겹쳐 확인해야 한다. "이거 진짜 동작하긴 해?"라는 의심이 이번엔 최고의 테스트였다.

## 레벨 생성 · 난이도 (Python)

### 난이도를 재는 봇은 똑똑함의 수준 자체가 지표를 좌우한다

레벨 난이도를 어떻게 정할까? 이 프로젝트는 봇에게 레벨을 여러 번 풀게 해서 클리어율(이긴 비율)로 티어를 매긴다. 그런데 봇이 얼마나 똑똑하냐에 따라 결과가 완전히 달라진다.

- 완벽한 봇(부을 수 있는 색을 늘 정확히 고르는 그리디 봇)은 거의 모든 레벨을 100% 깬다 → 쉬운 레벨과 어려운 레벨이 똑같이 100%라 구분 불가
- 완전 무작위 봇은 어려운 레벨을 0% 깬다 → 이것도 다 0%라 구분 불가

양극단 다 "어려움 = 클리어율 40~65%" 같은 밴드(구간)를 나누지 못한다. 필요한 건 평균적인 사람처럼 대체로 잘하다 가끔 실수하는 봇이다. 그래서 ε-greedy(엡실론 그리디)를 썼다. 이름 그대로 "대부분은 greedy(최선)하게, ε(작은 확률)만큼은 아무렇게나" 두는 방식이다. 여기선 ε=0.30~0.35, 즉 한 수마다 30~35% 확률로 최선 우선순위를 잊고 아무 바스켓이나 고른다.

```python
# ε-greedy: 한 수마다 35% 확률로 우선순위를 잊는다 — 평균 플레이어 모델
if rng.random() < BOT_MISTAKE_RATE:
    pool = options                                        # 실수: 아무거나
else:
    pourable = [bi for bi in options if has_reachable(color_of(bi))]
    pool = pourable if pourable else options              # 최선: 부을 수 있는 색 먼저
```

측정하다 알게 된 부수 사실 둘. 클리어율을 좌우하는 가장 큰 지렛대는 벽 배치가 아니라 바스켓을 몇 조각으로 쪼갰나(레일 슬롯 수 대비 얼마나 붐비나)였다. 그리고 색이 2개뿐인 도안은 어떤 배치로도 잘 안 어려워진다 — 색이 층층이 교차하는 일이 없어 "부을 데가 없어 기다리는" 상황이 드물어서다.

### 매장 깊이(peel depth) — 시뮬레이션 없이 구조 난이도를 계산한다

클리어율 측정은 봇을 수십 번 돌려야 해서 느리다. 그래서 훨씬 싼 1차 필터를 하나 뒀다. peel은 "껍질을 벗기다"라는 뜻인데, 여기선 "지금 꺼낼 수 있는 바스켓을 전부 걷어내는" 한 겹을 말한다.

매장 깊이(peel depth)는 보드가 빌 때까지 이 걷어내기를 몇 번 반복해야 하는지 센 값이다. 깊이 1이면 지금 있는 걸 한 번에 다 꺼낼 수 있는 얕은 배치, 깊이 5면 다섯 겹을 차례로 파내야 하는 깊이 묻힌 배치. 이 숫자가 클수록 구조적으로 더 까다롭다. 클리어율을 재는 것보다 비교도 안 되게 싸서, 이걸로 먼저 거른 뒤 밴드에 든 후보만 비싼 클리어율 측정으로 넘긴다.

```python
def peel_depth(walls, basket_cells):
    cells, depth = set(basket_cells), 0
    while cells:
        cells -= selectable(walls, cells)  # 지금 꺼낼 수 있는 것들을 한 겹 벗긴다
        depth += 1
    return depth
```

선택 규칙상 매 라운드 최소 1개는 반드시 걷어낼 수 있어서(빈 보드가 출구까지 연결돼 있다는 게 보장되면) 이 루프는 언젠가 끝난다.

### 클리어율은 저해상도 프로브로 잰다

같은 도안이라도 캔버스가 64×64면 클리어율을 한 번 재는 데도 오래 걸리고, 티어를 정하려면 이걸 여러 번 반복해야 해서 레벨 하나에 분 단위가 든다. 그런데 난이도를 만드는 건 퍼즐 구조(색이 몇 층으로 얽혔나, 바스켓 비율, 보드 배치)지 픽셀 개수가 아니다. 구조는 해상도를 낮춰도 그대로다.

그래서 도안을 32×32로 줄인 프로브(probe, 측정용 축소판) 캔버스에서 클리어율을 빠르게 재고(그레인 수가 1/4이라 훨씬 빠름), "이 레벨이 진짜 깰 수 있긴 한가"의 최종 보장만 풀 해상도로 그리디 봇 1회 돌려 확인했다. 정밀한 수치가 목적이 아니라 밴드로 분류만 하면 될 때는 해상도를 깎아도 된다.

### 벽 패턴은 만든 뒤 연결성부터 검증한다

빗살(위/아래)·주머니·기둥 4종 패턴으로 벽을 무작위로 깐다. 문제는 벽을 잘못 깔면 고립된 칸이 생긴다는 것 — 사방이 벽으로 막혀 출구까지 길이 없는 칸. 여기 바스켓이 들어가면 영영 못 꺼내는, 클리어 불가 레벨이 된다.

그래서 벽을 깐 뒤 flood-fill(홍수처럼 번지며 이어진 칸을 전부 방문하는 탐색)로 출구에서부터 번져서, 벽이 아닌 모든 칸에 닿는지 확인한다. 하나라도 못 닿으면(고립) 그 배치는 버리고 다시 만든다. 이렇게 "패턴 생성"과 "유효성 검증"을 분리해두면, 패턴 쪽은 유효성 걱정 없이 마음껏 무작위로 만들 수 있다.

```python
def open_cells_connected(w, h, walls):
    seen = flood_from_top(w, h, walls)          # 출구에서 번져 닿는 칸들
    return len(seen) == w * h - len(walls)       # 벽 뺀 전부에 닿았나
```

### 아래쪽에 깊이 묻으려면 — 가중치 비복원 추출

바스켓을 보드 아래쪽에 깊이 묻을수록 어려워진다(위에 것들을 먼저 치워야 꺼낼 수 있으니까). 그래서 바스켓 자리를 뽑을 때 아래 행일수록 뽑힐 확률이 높게 만들고 싶었다.

이걸 가중치 비복원 추출로 했다. "가중치"는 후보마다 뽑힐 확률에 차등을 주는 것(아래 행일수록 큰 가중치), "비복원"은 한 번 뽑은 자리는 후보 풀에서 빼고 다음을 뽑는 것(같은 자리 두 번 안 뽑히게). 이 둘을 반복하면 아래쪽이 잘 채워진 배치가 나온다. 얼마나 아래로 쏠리게 할지는 편향 계수(bias)로 조절하는데, 이게 티어 파라미터라 튜토리얼은 0(모든 행 균등), 어려움은 2.5~3.0(아래로 강하게 쏠림)으로 준다.

```python
# row가 클수록(아래일수록) 가중치가 커진다. bias가 쏠림 강도.
weights = [1.0 + bias * (row / (h - 1)) for (_, row) in pool]
```

### 비율을 정수로 나눌 때 합을 지키기 — 최대잉여법

한 색의 모래 총량을 여러 바스켓에 비율대로 나눠 담아야 한다. 예를 들어 총 100그레인을 33.3% / 33.3% / 33.3%로 나눈다고 하자. 그냥 각각 반올림하면 33+33+33=99가 되어 1그레인이 증발한다. 이 게임은 색별 총량이 캔버스와 딱 맞아야 클리어되므로, 1그레인만 어긋나도 클리어 불가가 된다.

그래서 최대잉여법(largest remainder)을 썼다. 방법은 두 단계다. (1) 먼저 각자 몫의 정수 부분(내림값)만 배분한다 → 33+33+33=99, 1 남음. (2) 남은 1을 소수 부분(0.3, 0.3, 0.3)이 큰 순서대로 하나씩 나눠준다. 이렇게 하면 합이 정확히 100으로 보존된다. 분할 크기에 변화를 주려고 가중치를 무작위로 흔들되, 각 바스켓이 최소 1은 받게 보장한다.

### 규칙의 불변식을 찾으면 탐색이 한 차원 준다 — 열 프론티어 캐시

용어 먼저. 불변식(invariant)은 "이 시스템에서 항상 참인 성질"이다. 코드가 아무리 돌아가도 절대 안 깨지는 규칙. 불변식을 하나 찾으면, 매번 전부 뒤지는 대신 그 성질을 믿고 계산을 건너뛸 수 있다.

여기 붓기 규칙은 이렇다 — 캔버스 픽셀은 (a) 맨 아래 행이거나 (b) 바로 아래 칸이 이미 채워졌을 때만 채워진다. 즉 모래는 중력으로 아래부터 쌓인다.

이 규칙에서 따라 나오는 불변식: 각 열(세로줄)의 채워진 칸은 항상 맨 아래부터 빈틈없이 연속이다(영어로 suffix — 뒤쪽/아래쪽 연속 구간). 중간이 비고 위만 찬 열은 절대 안 생긴다. 아래가 안 찼으면 그 위는 애초에 채울 수가 없으니까.

그래서 "지금 이 색을 부을 수 있는 칸"을 찾을 때, 열마다 후보는 딱 하나 — 채워진 부분 바로 위 한 칸(프론티어)뿐이다. 그 아래는 이미 찼고, 그 위는 아직 못 채우니까.

쉽게 말하면, 모래 한 알마다 캔버스 전체를 뒤지던 걸 가로 줄 개수만 보게 바꿔 수십 배 빨라졌다. 이 사실을 쓰기 전엔, 모래 한 알 부을 때마다 캔버스 전체(가로 W × 세로 H 칸)를 훑어 후보를 찾았다 — 훑는 양이 가로 × 세로에 비례(O(W×H)). 불변식을 쓰면 열마다 "지금까지 몇 칸 찼는지(height)"만 배열로 들고 있으면 되고, 프론티어는 그 높이에서 바로 계산된다. 훑을 게 열 개수뿐이라 가로에만 비례(O(W))하고, 세로 차원(H)이 통째로 사라진다. 난이도 측정처럼 시뮬레이션을 수십 번 돌릴 때 이 차이가 분 단위 대 초 단위로 벌어진다.

```python
# 열별 "채운 높이(heights[col])"만 유지한다.
# 그 열에서 다음에 채울 칸(프론티어) = 아래에서 heights[col]칸 위 = 행 (h - 1 - heights[col])
for col in range(w):
    row = h - 1 - heights[col]
    if target[row * w + col] == color and row > best_row:
        best_row, best_i = row, row * w + col  # 이 색을 부을 최선의 칸
```

"이렇게 줄여도 원래랑 결과가 같은가"의 동치는 머리로 증명(불변식이 보장)하고, 실제 검증은 기존 fixtures 테스트가 그대로 통과하는 걸로 확인했다. 규칙에 숨은 불변식을 찾으면 알고리즘 한 차원이 공짜로 빠진다.

## Flutter 연출

### 경로 애니메이션의 경로는 상태를 바꾸기 전에 계산한다

탭한 바스켓이 통로를 걸어 출구로 나가는 연출이 있다. 이때 엔진의 BFS 최단 경로(너비 우선 탐색으로 구한 가장 짧은 길)를 쓴다. 함정은 순서다. 바스켓을 보드에서 먼저 빼버리면, 그 자리가 빈칸이 돼서 "출발 칸에서 출구까지" 경로를 못 구한다(출발점이 사라졌으니까). 그래서 경로를 먼저 구하고, 그다음에 상태를 바꾼다.

```dart
final path = _state.pathToExit(boardIndex); // 빼기 전에 계산
final b = _state.takeFromBoard(boardIndex);
_walkers.add(_Walker(b, points: toOffsets(path)));
```

걷는 진행도는 t를 0부터 (구간 수)까지 흐르게 두고, 매 프레임 `t += dt × 속도`. 지금 위치는 `Offset.lerp(points[i], points[i+1], t - i)` — i번째와 i+1번째 지점 사이를 소수부만큼 선형 보간(lerp)해 얻는다. 선형 보간은 두 점 사이를 정해진 비율만큼 섞어 그 중간 지점을 잡는 것이다.

### 벨트 트레드는 공유 위상 하나로 굴린다

위상(phase)은 "지금 주기의 어디쯤인가"를 0~1로 나타낸 값이다(0에서 시작해 1이 되면 다시 0으로). 벨트 위 바스켓들은 각자 위치를 따로 갖는 게 아니라, 하나의 공유 위상(offset)을 두고 슬롯 s는 `offset + s/N` 위치에 있다(N=슬롯 수). 모두 같은 offset을 밀며 1/N 간격으로 줄지어 있으니, 구조적으로 서로 겹칠 수가 없다. 트레드(벨트 표면 빗금 무늬)도 같은 offset으로 스크롤하면, 바스켓과 벨트 바닥이 한 몸처럼 같이 흘러가 보인다.

```dart
final shift = (offset * size.width) % spacing;
for (var x = -spacing + shift; x < size.width + spacing; x += spacing) {
  canvas.drawLine(Offset(x, size.height), Offset(x + slant, 0), stripe);
}
```

CustomPainter가 다시 그릴지 판단하는 `shouldRepaint`는 이 위상이 바뀌었는지만 보면 된다.

### 모래 줄기는 2차 베지어 곡선 + 위상 순환 입자

붓는 바스켓에서 캔버스 목표 픽셀로 모래가 주르륵 떨어지는 줄기 연출. 곡선은 2차 베지어(quadratic bezier) — 실로 매단 두 점 사이를 세 번째 점이 당겨 만드는 부드러운 곡선이다. 시작점, 끝점, 그리고 곡선을 당기는 제어점(control point) 하나로 정의된다. 제어점을 시작점의 x에 두면 처음엔 거의 수직으로 떨어지다가 끝점 쪽으로 휘어진다.

입자(모래알)는 곡선 위를 흐르게 하는데, 매개변수 u(0=시작, 1=끝)를 `(k/개수 + 위상) % 1`로 돌린다. 위상이 조금씩 커지며 모든 입자가 곡선을 따라 밀려가고, 1을 넘으면 다시 0으로 감싸여(% 1) 끊김 없이 흐른다.

```dart
final ctrl = Offset(from.dx, from.dy + (to.dy - from.dy) * 0.6);
for (var k = 0; k < grains; k++) {
  final u = ((k / grains) + phase) % 1.0;
  canvas.drawCircle(bezier(from, ctrl, to, u), 1.8 + 1.2 * u, dot);
}
```

목표 픽셀은 엔진의 "지금 이 색이 부어질 위치" 조회를 그대로 써서, 연출과 실제 게임 상태가 어긋나지 않는다.

### Path.combine(difference)로 스포트라이트를 뚫고, 터치는 따로 강제한다

튜토리얼에서 화면 전체를 어둡게 덮되 눌러야 할 대상만 밝게 뚫어주는 오버레이. 이걸 도형 뺄셈으로 만든다. 화면 전체 사각형 Path에서, 구멍 낼 자리(둥근 사각형)들을 `Path.combine(difference)`(앞 도형에서 뒤 도형을 도려내는 연산)로 빼면, 구멍 뚫린 어둠 한 장이 한 번에 그려진다.

```dart
var path = Path()..addRect(Offset.zero & size);
for (final h in holes) {
  path = Path.combine(PathOperation.difference, path,
      Path()..addRRect(RRect.fromRectAndRadius(h, Radius.circular(r))));
}
canvas.drawPath(path, Paint()..color = const Color(0x8C000000));
```

그런데 밝게 뚫었다고 그쪽이 눌리는 건 아니다 — 그리기와 터치 판정은 별개다. 그래서 오버레이 위젯을 `GestureDetector(behavior: HitTestBehavior.opaque)`로 감싸 모든 터치를 빨아들이고, 눌러야 할 대상 위젯만 오버레이보다 위에 다시 얹어(Stack에서 뒤쪽 순서 = 화면 앞쪽) 그쪽 탭만 통하게 한다. Stack의 children 순서가 곧 z-order(앞뒤 겹침 순서)라는 점은 벨트 하우징에도 써먹었다 — 장착된 바스켓은 하우징 밑으로 지나가고, 대기 중 바스켓은 위로 올라와 강조되게.

### 잔량 게이지는 FractionallySizedBox 하나면 된다

바스켓(양동이) 안이 남은 모래 비율만큼 차 있는 표시. 별도 페인터를 짤 것 없이, 바닥 정렬한 `Align` 안에 `FractionallySizedBox`(부모 크기의 몇 배로 자식을 잡는 위젯)를 넣고 `heightFactor`에 채움 비율만 주면 끝난다.

```dart
Align(
  alignment: Alignment.bottomCenter,
  child: FractionallySizedBox(
    heightFactor: fill.clamp(0.0, 1.0),
    widthFactor: 1,
    child: ColoredBox(color: sandColor),
  ),
)
```

부모 Container에 `clipBehavior: Clip.antiAlias`를 줘야 채움이 둥근 모서리 밖으로 삐져나오지 않는다.

### append-only 델타 원장은 필드를 안 늘리고 개념을 늘린다

원장(ledger)은 회계 장부처럼 "변화(델타)를 시간순으로 append(뒤에 덧붙이기)만 하고, 지우거나 고치지 않는" 기록 방식이다. 코인·하트 같은 재화의 현재 값을 필드로 저장하는 대신, `+50 코인`, `-1 하트` 같은 변화 레코드를 쭉 쌓아두고 전부 접어서(fold, 처음부터 다 더해서) 현재 값을 도출한다.

이 구조의 장점은 새 개념이 생겨도 스키마(데이터 구조)를 안 바꿔도 된다는 것. 아이템 인벤토리(`item_undo` 증감)도, 온보딩 보상 지급 이력(`grant` = 준 레벨 id 표시)도, 새 kind 문자열 하나만 추가하면 끝이다. "이 레벨 보상 이미 줬나?"도 "grant 델타가 있나?" 조회로 해결된다.

```dart
int itemCount(String item) => _sum('item_$item');          // 그 kind 델타를 다 접기
bool hasGrant(int levelId) =>
    _deltas.any((d) => d.kind == 'grant' && d.value == levelId);
```

### 위젯 테스트의 pump는 가짜 시계다 — 실시간 게임 루프도 E2E로 돈다

이 게임은 실기기 없이 데스크톱 툴체인만으로는 `flutter run`이 안 되는 환경이었다. 그래도 실시간 루프(Ticker = 매 프레임 콜백을 부르는 타이머) 위의 전체 흐름(탭 → 경로 걷기 → 엘리베이터 → 벨트 장착 → 붓기 → 완성 오버레이)을 검증하고 싶었다.

여기서 핵심은 `tester.pump(Duration)`이 진짜 시간을 안 기다리고 "가짜 시간"을 그만큼 흘려보낸다는 것. pump 한 번에 Ticker가 그 시간만큼 틱을 돈다. 그래서 탭한 뒤 pump를 여러 번 돌리고 나서 완성 오버레이 텍스트를 expect하면, 그 사이 파이프라인 전체가 예외 없이 끝까지 갔는지(E2E, 처음부터 끝까지) 확인된다.

```dart
await tester.tap(find.text('100%'));
for (var i = 0; i < 40; i++) {
  await tester.pump(const Duration(milliseconds: 100)); // 가짜 시간 4초어치
}
expect(find.text('완성! 🎉'), findsOneWidget);
```

기기에서 직접 돌려볼 수 없는 환경에서, 런타임 회귀를 잡는 대체 수단으로 쓸 수 있다.
