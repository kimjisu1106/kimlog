## 레벨 파이프라인 (Python)

### 이모지 그림을 색 줄여 32×32 도안으로 변환한다

손으로 도형을 계속 그릴 순 없어서 이미지에서 도안을 뽑는 파이프라인을 만들었다. 오픈소스 이모지(Noto, Apache 2.0)를 32×32 팔레트 인덱스 그리드로 바꾼다. 양자화는 사진 속 수많은 색을 대표하는 몇 색으로 줄이는 것으로, 그림에 맞춰 대표색을 고르므로 "적응(adaptive)" 양자화라 부른다.

핵심은 "배경을 먼저 지우고 전경만 양자화"하는 것. 투명/배경을 그대로 두면 양자화가 배경용 가짜 색 슬롯을 하나 잡아먹는다. 그래서 배경 픽셀을 전경 평균색으로 채운 뒤 양자화하고, 배경(index 0)은 나중에 주조색의 옅은 틴트로 따로 만든다.

```python
rgb = ...  # 배경 픽셀을 전경 평균색으로 채운 이미지
q = rgb.quantize(colors=MAX_COLORS, method=Image.Quantize.MEDIANCUT)  # 최대 5색
# 클러스터별 픽셀 모으기
clusters = {}
for x, y in fg:
    clusters.setdefault(qpx[x, y], []).append((x, y))
```

그다음 슬롯 낭비를 막는 두 단계 병합이다.

- 1% 미만 미세 클러스터는 최근접 대표색으로 흡수 (한두 픽셀짜리 색이 팔레트 한 칸을 먹지 않게)
- 거의 같은 색끼리도 병합 — 작은 쪽을 큰 쪽에 합치고, 대표색은 픽셀 수 가중 평균

이렇게 "비배경 최대 5색 + 배경 틴트"로 도안당 색 수를 게임에 맞게 눌러 담는다. 라이선스는 소스 폴더에 LICENSE를 동봉했다(Apache 2.0).

### 수식으로 32×32 절차 도형을 그린다

이모지만으로는 부족해서 절차 도형 10종(하트·별·버섯·꽃·링 등)도 8×8 ASCII 템플릿을 버리고 수식 기반 32×32로 다시 그렸다. 중심 거리·각도로 형태를 정하고, 안쪽/바깥쪽 밴드에 셰이딩·하이라이트를 얹어 도형당 3~5색을 만든다. 캔버스 해상도(64×64)는 그대로지만 도안의 실질 디테일이 4×4배로 올라간다.

### 도안 자체가 난이도의 절반 — 색 레이어 교차도로 점수화

이모지마다 봇 클리어율이 0%(나선 사탕처럼 색이 얽힌 것)에서 94%(단색 얼굴)까지 널뛰었다. 도안이 이미 난이도를 상당히 결정해버린다. 티어에 무작정 배정하면 같은 티어인데 체감이 딴판이 된다.

그래서 도안의 "색 레이어 교차도"를 점수로 뽑았다 — 도형이 있는 열마다 위에서 아래로 색이 몇 번 바뀌는지의 평균. 같은 색이 여러 높이에 흩어질수록(교차가 많을수록) 붓기 대기가 자주 나 어렵다는 §6 직관을 수치화한 것. 이 점수로 티어 버킷에 배정한다.

교차도는 어디까지나 근사치라, 안전망을 뒀다 — 실제 측정 클리어율이 목표 밴드에서 0.15 이상 벗어난 도안은 버리고 다시 뽑는다. 밴드 정의(`RATE_BANDS`)와 티어 판정(`tier_for`)을 함수로 빼서, 레벨 계획과 도안 배정이 같은 기준을 공유하게 했다.

### 픽셀 적은 색이 안 보이던 문제 — 최소 모래량 하한

색을 여러 바스켓으로 쪼갤 때, 픽셀이 적은 색은 잘게 나뉘어 한 바스켓이 붓혀도 캔버스에 티가 안 났다. 바스켓 1개가 붓는 양에 하한(64×64 기준 110그레인)을 둬서, 그보다 잘게 쪼개지지 않게 했다.

### 측정은 저해상도, 클리어 보장은 풀 해상도

티어를 정하려면 도안마다 클리어율을 여러 번 재야 하는데 풀 해상도(64×64)는 느리다. 퍼즐 구조(색 레이어·순서)는 해상도와 무관하므로, 측정은 32×32 프로브 캔버스로 빠르게 돌리고, "이 레벨이 진짜 클리어 가능한가"의 최종 보장만 풀 해상도로 확인한다.

## 연출 (Flutter)

### 밝기 지터가 8px 줄무늬를 만든 이유 — 숫자를 잘 섞어 쓰지 않아서

모래 알갱이 질감을 주려고 픽셀 인덱스로 밝기를 살짝 흔들었는데(TIL 2), 화면에 8픽셀 주기의 은은한 명암 경사(줄무늬)가 생겼다. 원인은 지터 값을 `(i * 48271) & 0x7`로 뽑은 것. 하위 3비트만 쓰면 인덱스가 1씩 늘 때 값이 0~7로 규칙적으로 순환하고, 캔버스 폭 방향으로 이 주기가 줄무늬로 보인다.

정리하면, 픽셀 번호를 그냥 쓰면 규칙적인 무늬가 화면에 새어 나온다. 그래서 번호를 한 번 잘 섞어 뒤죽박죽으로 만든 뒤 밝기로 쓴다. 곱셈 해시의 하위 비트는 상위 비트보다 무작위성이 약하므로, 하위 비트만 떼지 말고 비트를 섞어(shift·xor·곱셈) 흩뜨린 뒤 써야 한다. 아래는 이 목적에 업계 표준으로 쓰이는 섞기 공식(MurmurHash 파이널라이저)이다.

```dart
var h = i * 0x9E3779B1;
h ^= h >> 15;
h = (h * 0x85EBCA6B) & 0x7FFFFFFF;
h ^= h >> 13;
// 이제 h의 하위 비트도 잘 섞여 있어 규칙적 순환이 사라진다
```

"인덱스 기반 결정적 값"은 좋지만, 그 값을 만들 때 하위 비트를 그대로 쓰면 격자무늬가 새어 나온다. 결정적이되 섞여 있어야 한다.

## 내비게이션 (Flutter)

### 화면을 갈아끼우면 원래 화면이 하려던 뒷정리를 놓친다 — 떠나는 순간에 처리

레벨을 깨고 "다음 레벨"로 넘어가는데, 목록으로 돌아오면 방금 깬 레벨의 클리어 표시가 안 떴다.

원인은 화면 전환 방식이다. 목록에서 레벨을 열 때 `await navigator.push(...)`로 띄우고 "이 화면에서 돌아오면 목록을 갱신"하도록 그 뒤에 갱신 코드를 뒀다. 그런데 "다음 레벨"은 `pushReplacement`(현재 화면을 새 화면으로 통째로 갈아끼우는 전환)로 현재 게임을 다음 게임으로 바꾼다. 화면을 A→B로 교체하면 원래 A 화면이 "돌아왔을 때 하려던 일"을 놓친다 — `push`의 await가 (화면이 교체됐으므로) 먼저 완료돼 버려, 정작 게임을 나가는 시점의 갱신을 놓치는 것.

해결은 "게임 화면이 실제로 닫히는(pop) 순간"에 직접 갱신하는 것. 방을 나가며 불을 끄듯, 떠나는 그 순간에 뒷정리를 한다. 게임 페이지를 `PopScope`(화면이 닫힐 때 콜백을 주는 위젯)로 감싸 그 콜백에서 목록을 갱신하면, HUD 뒤로가기·완성 오버레이 버튼·시스템 뒤로가기 어느 경로로 나가도 클리어 표시가 반영된다.

```dart
Widget _wrapGame(int index) => PopScope(
  onPopInvokedWithResult: (didPop, _) {
    if (didPop && mounted) setState(() {}); // 나가는 순간 목록 갱신
  },
  child: GamePage(/* ... onNext: pushReplacement 체인 ... */),
);
```

`await push` 뒤에 후처리를 두는 패턴은 pushReplacement 체인이 끼면 깨진다. "돌아온 뒤"가 아니라 "나가는 순간"(pop 이벤트)에 반응해야 안전하다.
