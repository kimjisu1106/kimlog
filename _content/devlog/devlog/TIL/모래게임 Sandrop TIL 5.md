## 스프라이트 착색·연출

### 색이 무한하면 스프라이트를 색별로 만들지 말고 한 장을 틴트한다

이 게임은 레벨마다 팔레트가 다르고(이모지 양자화 도안은 도안별 전용 팔레트) 바스켓 색이 사실상 무한하다. 양동이 스프라이트를 색마다 그리는 건 불가능하다. 그래서 밝은 크림톤 원본 한 장을 두고 코드에서 색을 곱해(modulate) 착색한다.

`BlendMode.modulate`는 두 색을 곱하는 블렌드다. 흰색(1.0)에 색을 곱하면 그 색이 그대로 나오고, 회색조의 명암은 곱해지면서 보존된다. 그래서 밝은 원본일수록 착색이 깔끔하다.

```dart
Image.asset(
  'assets/images/bucket-topdown.png',
  color: Color.lerp(sandColor, Colors.white, 0.22), // 살짝 밝혀 명암 살림
  colorBlendMode: BlendMode.modulate,
  fit: BoxFit.contain,
)
```

한 장으로 236종 도안의 모든 색을 커버한다. 막힘(디밍)도 오버레이를 덮지 않고 틴트 색 자체를 회색 쪽으로 눌러 표현하니 더 깔끔했다.

### 담긴 모래는 원반 + 하이라이트 + 그레인 점을 레이어로 쌓는다

양동이 안이 "모래로 수북한" 인상은 단색 원 하나로는 안 나온다. 개구부 안에 Stack으로 세 겹을 쌓았다 — 본색 원반, 위쪽에 밝은 하이라이트 소타원(빛 받는 면), 인덱스 해시로 흩뿌린 어두운 그레인 점 3개. 잔량 비율에 따라 원반 크기가 커지고 줄어든다.

### 통 안 수위는 사각 채움이 아니라 원으로 — 면적 비례 sqrt

우측 대기 칩(원형)에 아래에서 차오르는 사각 채움을 넣었더니, 부분 채움일 때 모래 윗면이 직선으로 잘려 각져 보였다. 원 안엔 원이 맞다. `FractionallySizedBox`로 중심에서 커지는 원을 그리되, 지름을 그냥 fill에 비례시키면 체감이 안 맞는다. 동그라미는 지름을 2배 키우면 넓이가 4배가 되기 때문이다. 그래서 절반 남은 것을 절반 넓이로 보이게 하려면 지름은 √0.5 ≈ 0.7배로 잡아야 한다. 사람 눈은 넓이로 양을 읽으므로 지름은 `sqrt(fill)`로 잡아야 넓이가 fill에 비례한다.

```dart
final f = math.sqrt(fill.clamp(0.0, 1.0)); // 넓이 비례
FractionallySizedBox(widthFactor: f, heightFactor: f, child: 원);
```

### 거꾸로 붓기 — 통은 뒤집되 라벨은 안에서 되돌린다

부을 수 있는 벨트 양동이는 모래를 쏟는 것처럼 상하 반전시킨다(`Transform.rotate(math.pi * tip)`). 통째로 돌리면 % 라벨까지 뒤집혀 읽을 수 없으므로, 라벨만 자식으로 한 번 더 역회전시켜 항상 정립하게 한다.

반전은 딱 끊지 않고 tipFrac(0..1)을 목표값으로 매 틱 감속 보간해 부드럽게 기운다. 부을 수 있으면 1로, 없으면 0으로 접근.

```dart
// 목표(부을 수 있으면 1)로 프레임마다 조금씩
final target = state.pourTargetOf(color) >= 0 ? 1.0 : 0.0;
tip += (target - tip).sign * dt * tipSpeed; // clamp로 목표 넘지 않게
```

## 신호·라벨 정책

### 강조는 양동이가 아니라 그 아래 바닥이 준다

"내보낼 수 있는 칸"을 양동이 자체 글로우로 표시했는데, 스프라이트가 반투명 여백을 가져 글로우가 지저분했다. 강조를 받침 패드(바닥 타일)로 옮기니 깔끔했다 — 보드에서는 바닥이 빛나 신호를 주고 양동이는 담백하게 둔다. 받침 바닥은 통로(빈칸)와 같은 파낸 굴 톤으로 통일해, 빈칸/바스켓 칸의 바닥이 한 재질로 읽히게 했다.

### 같은 % 라벨도 문맥마다 기준이 다르다

바스켓 잔량 %는 원래 "색 총량 대비"였다. 벨트에서 부으며 줄어드는 걸 보여줄 땐 이 기준이 헷갈린다 — 플레이어가 궁금한 건 "이 통이 얼마나 남았나"지 색 전체가 아니다. 그래서 벨트에서는 "그 양동이가 처음 담은 양 = 100%" 기준으로 바꾸고(초기량을 통 id별로 기록), 보드에서는 아예 % 를 숨겼다. 같은 숫자라도 어디서 보여주느냐에 따라 분모를 바꿔야 의미가 산다.

### 경로 잔상은 위젯 말고 페인터 하나에 모은다

바스켓이 통로를 걸어 나갈 때 잔상(트레일)을 남긴다. 잔상 점마다 위젯을 만들면 프레임마다 위젯이 수십 개 생겼다 사라진다. 워커 위치를 일정 간격으로 샘플링해 (위치, 생성시각) 리스트에 쌓고, 이미 있던 모래 줄기 CustomPainter에 파라미터로 넘겨 한 곳에서 나이에 따라 작아지며 페이드되게 그렸다. 잔상·줄기처럼 "많고 짧게 사는" 연출은 위젯보다 페인터가 맞다.

## 규칙과 UI의 연결

### 공개 "시점"은 규칙이다 — 화면만 고치면 안 된다

물음표(색 숨긴 바스켓)를 넣었더니 자리에 따라 시작하자마자 색이 열렸다. 원인은 "선택 가능해지는 순간 공개"였는데, 데뷔 레벨의 물음표가 시작부터 선택 가능한 자리에 있으면 렌더 첫 프레임에 공개돼버린 것.

이건 화면 버그가 아니라 규칙 문제라, 공개 시점을 "보드를 떠나 엘리베이터에 오르는 순간"으로 바꿨다. 그러면 배치와 무관하게 보드에 있는 동안(걸어 나가는 중 포함) 계속 `?`다. 구현상 주의점: 선택 판정 함수(`selectablePositions`)가 공개 부작용을 갖고 있어서, 로프 짝 해방 가정 검사처럼 "가정만 해보는" 호출에서도 색이 공개돼버렸다. 공개 부작용을 판정에서 떼어 "보드 이탈(`takeFromBoard`)" 한 곳으로 모았다.

### 규칙을 바꾸면 그 규칙을 읽던 화면 코드가 같이 썩는다

공개 시점을 바꿨더니, 옛 규칙 기준으로 "누르면 안 되는 칸"으로 그리던 화면 코드가 그대로 남아 물음표가 길이 열려도 안 눌렸다. 보드 렌더 분기가 옛 규칙(공개 전엔 조작 불가) 기준이라, 미공개 `?` 타일을 GestureDetector(탭을 받는 위젯) 없이 항상 비활성으로 그리고 있었다. 규칙(공개 시점)만 고치고 그 규칙을 소비하던 UI 분기를 안 고친 것.

교훈은 규칙 변경의 영향 반경을 끝까지 따라가라는 것. "공개는 언제?"를 바꾸면 "그럼 공개 전/후 렌더는?", "탭 가능 여부는?"까지 딸려온다. 미공개/공개 분기를 합쳐 탭·디밍·바닥 글로우 로직을 공유하고, 차이는 타일 색·라벨(`?`)뿐으로 정리했다.

## 빌드·테스트

### adaptive 아이콘은 전경·배경을 분리해 넣는다

앱 아이콘을 도안(고래)으로 만들 때 `flutter_launcher_icons`를 썼다. 안드로이드 adaptive 아이콘은 전경(도형)과 배경(단색/이미지)을 분리해 지정하면, 런처가 기기 마스크(원·둥근사각 등)에 맞춰 알아서 잘라준다. `image_path`(레거시 정사각)와 `adaptive_icon_foreground`/`adaptive_icon_background`를 함께 주고 한 번 실행하면 여러 화면 크기용 이미지(mdpi~xxxhdpi 밀도 묶음)가 한 번에 일괄 생성된다.

```yaml
flutter_launcher_icons:
  android: true
  image_path: "assets/icon/icon_master_1024.png"
  adaptive_icon_background: "#0C3B4C"
  adaptive_icon_foreground: "assets/icon/icon_adaptive_fg_1024.png"
```

### 새 에셋을 넣고 테스트가 깨지면 테스트 에셋 번들을 의심한다

양동이 PNG를 넣고 위젯 테스트를 돌리니 "Unable to load asset"으로 깨졌다. pubspec에 등록도 했는데. 원인은 `build/unit_test_assets/`에 예전 에셋 매니페스트가 캐시로 남아 있던 것 — 새 에셋이 번들에 안 들어갔다. 그 폴더를 지우고 다시 돌리니 통과했다. 에셋 관련 테스트 실패는 코드보다 스테일 번들을 먼저 의심할 것.
